# Auto-instrumentation

New integrations must support compile-time auto-instrumentation, so a user gets tracing without
editing their code. Today this is delivered with Orchestrion, described below. This guide covers how
to add auto-instrumentation to an integration. For building the integration itself, see
[INTEGRATIONS.md](./INTEGRATIONS.md).

## Orchestrion

Orchestrion rewrites the Go AST at build time using an `orchestrion.yml` file in the integration.
References: the [Orchestrion contributor guide](https://datadoghq.dev/orchestrion/contributing/aspects/),
the [in-repo overview](../orchestrion/README.md), and the existing `contrib/*/orchestrion.yml` files.

### orchestrion.yml

An `orchestrion.yml` contains aspects. Each aspect is a join point (where to act) and advice (what to
inject), rendered with Go `text/template`.

For a hook, middleware, or interface wrapper, wrap the constructor call so it still returns the same
value:

```yaml
aspects:
  - id: NewClient
    join-point:
      function-call: github.com/redis/go-redis/v9.NewClient
    advice:
      - wrap-expression:
          imports:
            redis: github.com/redis/go-redis/v9
            trace: github.com/DataDog/dd-trace-go/contrib/redis/go-redis.v9/v2
          template: |-
            func() (client *redis.Client) {
              client = {{ . }}
              trace.WrapClient(client)
              return
            }()
```

For a constructor replacement, swap the call one-to-one:

```yaml
  - id: sql.Open
    join-point:
      function-call: database/sql.Open
    advice:
      - replace-function: github.com/DataDog/dd-trace-go/contrib/database/sql/v2.Open
```

Join points used across the repo: `function-call`, `function-body` (matched by receiver, name, or
signature), `struct-definition`, `struct-literal`, and the combinators `one-of`, `all-of`, and
`not`. Advice used: `wrap-expression`, `replace-function`, `append-args`, `prepend-statements`,
`add-struct-field`, `inject-declarations`. To avoid a circular import back into your own contrib, use
`not: import-path: <self>` or `go:linkname`.

## Integration tests are mandatory

Auto-instrumentation is only proven by the tests under
[internal/orchestrion/_integration](../internal/orchestrion/_integration). See its
[README](../internal/orchestrion/_integration/README.md).

- A test is a `TestCase` with `Setup`, `Run`, and `ExpectedTraces`.
- The application code imports the target library and never imports dd-trace-go. Spans must appear
  only from the weave.
- Registration is generated by `go generate ./...`.
- Write one file per distinct calling convention the library supports: function literal, interface,
  closure, global convenience function, explicit construction, value versus pointer config. This
  exercises the join-point matchers and catches build breaks at compile time. See
  `net_http/issue_400.go` and `net_http/global_functions.go`.

After editing any `orchestrion.yml`, run `make generate` so `orchestrion/all` and the generated tests
stay in sync, or CI fails.
